--------------------------------------------------------------------------------
A. Authentication & Security Mapping Tables
--------------------------------------------------------------------------------

- user_roles: Connects auth.users accounts to app-specific roles.
  * id (UUID, PK) -> References auth.users(id) ON DELETE CASCADE
  * role (TEXT) -> Defaults to 'user', supports 'admin'

- admins: Configured list of administrator emails.
  * id (UUID, PK) -> References auth.users(id) ON DELETE CASCADE
  * email (TEXT) -> Admin user's email address

--------------------------------------------------------------------------------
B. Academic & Student Ledger Tables
--------------------------------------------------------------------------------

- students: Master catalog of students.
  * id (UUID, PK) -> Defaults to gen_random_uuid()
  * student_id (TEXT, UNIQUE) -> Custom identification string (e.g. AN-123456)
  * name (TEXT) -> Full student name
  * phone (TEXT) -> Contact number
  * email (TEXT, UNIQUE, NULLABLE) -> Unique student email address
  * batch (TEXT) -> Batch option (Repeater, Weekend Batch, Crash Course, etc.)
  * username (TEXT) -> Used for student-specific login
  * password (TEXT) -> Password corresponding to student username
  * place (TEXT) -> Hometown location
  * month_of_joining (TEXT) -> Record of active join period
  * status (TEXT) -> Active or Inactive status

- tests: Schedules and catalogs for conducted academy examinations.
  * id (UUID, PK) -> Defaults to gen_random_uuid()
  * name (TEXT) -> Test description/title
  * date (DATE) -> Day the test was taken
  * type (TEXT) -> Category classification

- student_marks: Detailed exam results.
  * id (UUID, PK) -> Defaults to gen_random_uuid()
  * student_id (UUID, FK) -> References students(id) ON DELETE CASCADE
  * test_id (UUID, FK) -> References tests(id) ON DELETE CASCADE
  * physics (NUMERIC) -> Marks scored in Physics
  * chemistry (NUMERIC) -> Marks scored in Chemistry
  * biology (NUMERIC) -> Marks scored in Biology
  * total (NUMERIC) -> Cumulative total (Sum of P+C+B, max 720)
  * percentage (NUMERIC) -> Performance percentage

--------------------------------------------------------------------------------
C. CMS & Configuration Tables
--------------------------------------------------------------------------------

- website_content: Fallback JSON data for miscellaneous CMS widgets.
  * key (TEXT, PK) -> Unique configuration identifier
  * content (JSONB) -> Structured payload values

- website_home: Homepage CMS configs. Single row where id = 'main'.
- website_about: Story, mission, vision elements. Single row where id = 'main'.
- contact_information: Social icons and map embeds. Single row where id = 'main'.
- courses: Curriculum card items.
- gallery: Media assets grid references.
- achievements: NEET toppers registry names, ranks, scores, and image references.

3. CLIENT & SERVER CONNECTION WORKFLOWS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The portal uses Next.js 15 App Router architecture. Database operations are
split between browser-side and server-side connections:

--------------------------------------------------------------------------------
A. Client-Side (Browser Client) -> src/lib/supabase/client.ts
--------------------------------------------------------------------------------
- Initialized using createBrowserClient() from @supabase/ssr.
- Reads environment variables NEXT_PUBLIC_SUPABASE_URL and
  NEXT_PUBLIC_SUPABASE_ANON_KEY.
- Implements a singleton instance check (supabaseInstance) to prevent duplicate
  listeners or connections.
- Used in page views and dashboards for SWR fetching, live state mutation,
  and admin editing interfaces.

--------------------------------------------------------------------------------
B. Server-Side (Server Client) -> src/lib/supabase/server.ts
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- Initialized using createServerClient() from @supabase/ssr.
- Integrates with Next.js cookies() context to retrieve, modify, and refresh
  authentication tokens.
- Used inside Server Actions, API routes, and Server Components.

4. AUTHENTICATION & SECURITY WORKFLOW
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

The portal implements a dual-login security system to accommodate both
staff administrators and academy students.

--------------------------------------------------------------------------------
A. Student Authentication Flow
--------------------------------------------------------------------------------
1. Student enters their credentials (name as username, mobile number as password) on `/login`.
2. The login page triggers a POST request to the secure backend API `/api/student/login`.
3. The API queries the database using a security definer RPC (`get_student_by_username`) to fetch details.
4. The server validates the entered mobile number against the `password_hash` using `bcrypt.compareSync()`.
5. If verified, the server:
   - Updates the last login timestamp (`update_student_last_login` RPC).
   - Generates a signed session token `{ student_id, expiresAt }` signed via HMAC-SHA256 (Web Crypto API).
   - Sets a secure, signed `student_session` cookie:
     `HttpOnly = true, Secure = true, SameSite = Lax, MaxAge = 24 Hours`.
6. The client is redirected to `/student/dashboard`.
7. The student dashboard loads details using `/api/student/dashboard-data` which reads the `student_session` cookie.

--------------------------------------------------------------------------------
B. Admin Authentication Flow
--------------------------------------------------------------------------------
1. Admin enters their email and password on `/login`.
2. Client sends request to GoTrue endpoint:
   `await supabase.auth.signInWithPassword({ email, password })`
3. Upon successful credentials match:
   - Supabase returns the User payload.
   - The client inspects data.user.user_metadata.role.
   - If not in metadata, queries public.user_roles where id = user.id.
4. If role is NOT 'admin', the client signs them out immediately via
   `supabase.auth.signOut()` and throws "Access Denied".
5. If role IS 'admin', access is granted and redirects to `/admin/dashboard`.

--------------------------------------------------------------------------------
C. Edge Middleware Guard Path (src/middleware.ts)
--------------------------------------------------------------------------------
- Next.js middleware compiles into a Netlify Edge Function.
- It matches ONLY specific routes to bypass public resources and avoid timeouts:
  * `/admin/dashboard/*`
  * `/student/dashboard/*`
  * `/login`
- For `/admin/dashboard/*`: If no session exists or user role !== 'admin', it
  redirects them back to `/login`.
- For `/student/dashboard/*`: If no `student_session` cookie is found or the signature is invalid/expired, it redirects them back to `/login`.

5. STORAGE & MEDIA PIPELINE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

All website assets (NEET topper pictures, campus gallery cards, course brochures)
are stored in a public Supabase Storage bucket named "academy".

Workflow:
1. Admin selects a local file on their machine.
2. File name is sanitized and prefixed with folder destination and timestamp:
   `const fileName = `${folder}/${Date.now()}_${sanitized_name}``
3. Browser client uploads the file directly to the bucket:
   `await supabase.storage.from("academy").upload(fileName, file)`
4. Client retrieves the public HTTP endpoint:
   `const publicUrl = supabase.storage.from("academy").getPublicUrl(fileName).data.publicUrl`
5. The publicUrl is saved into the corresponding table column (e.g. courses.image_url).

6. PERFORMANCE & SYNCHRONIZATION (SWR & OPTIMISTIC UI)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

To ensure instantaneous loading times and zero UI latency, the CMS implements
a custom Stale-While-Revalidate caching pattern:

1. **Mounting**: Component checks localStorage for cached content.
2. **Interactive UI**: If cache exists, it loads instantly into state.
3. **Background Validation**: Component triggers a silent query to Supabase.
4. **State Reconciliation**: If data changed, the state updates and updates the
   localStorage cache silently. A pulsing "Syncing" badge is shown in the header.
5. **Optimistic Mutations**: On save/delete, local React state and localStorage cache
   mutate immediately. The server call runs in the background. If the API returns
   an error, the UI state and cache roll back to the original database snapshot.
